# Аудит `react-hooks/exhaustive-deps` подавлений

> **Контекст:** React Compiler требует, чтобы все компоненты и хуки строго следовали [Rules of React](https://react.dev/reference/rules).
> Подавление `eslint-disable-next-line react-hooks/exhaustive-deps` сигнализирует компилятору, что зависимости
> хука не соответствуют его телу — компилятор может сгенерировать некорректный код мемоизации или полностью
> пропустить оптимизацию для этого компонента.

## A. Stable ref (Reanimated SharedValue)

## C. Mount-only effect

**Суть проблемы:** `useEffect(..., [])` — намерение выполнить код один раз при маунте. Но тело
замыкает переменные, которые теоретически могут измениться.

### C1. `use-sync-settings.tsx:46`

```
src/hooks/setup/use-sync-settings.tsx
```

```tsx
useEffect(() => {
	// Читает theme, colorScheme
	// Вызывает setSettingsValue, Appearance.setColorScheme
	// Подписывается на SettingsBridgeModule.addListener
	return () => {
		SettingsBridgeModule.removeAllListeners('onThemeChanged');
	};
	/* eslint-disable-next-line react-hooks/exhaustive-deps */
}, []);
```

- **Пропущены:** `theme`, `colorScheme`
- **Проблема:** Эффект делает две вещи — подписка (mount-only) и синк темы (реактивный).
  Если `theme` сменится после маунта — эффект не среагирует.
- **Решение:**
  1. Разделить на два `useEffect`: один — подписка на `onThemeChanged` с `[]`, второй — реакция на `theme`/`colorScheme`.
  2. Подписку можно обернуть в `useRef`-guard или вынести в отдельный хук.

### C2. `use-star-animations.ts:46`

```
src/screens/settings/components/app-logo-picker/animated-star/use-star-animations.ts
```

```tsx
useEffect(() => {
    glowScale.value = withRepeat(withSequence(...), -1, true);
    /* eslint-disable-next-line react-hooks/exhaustive-deps */
```

- **Решение:** `glowScale` — stable Reanimated ref. Просто добавить в deps. Фактически это

## D. `.length` вместо массива

**Суть проблемы:** `useMemo` зависит от `transactions.length` вместо `transactions`, чтобы не
пересчитывать при смене ссылки на массив, если длина не изменилась.

**Почему опасно:**

- Если массив поменялся по содержимому (например, элемент обновился), но длина осталась — memo вернёт stale данные.
- React Compiler **полностью игнорирует** ручные deps и мемоизирует по своим правилам. Если компилятор
  решит, что нужно пересчитать — он это сделает. Если нет — нет. Ручной `.length` хак будет конфликтовать.

### D1. `use-summaries.ts:81`

```
src/screens/transactions/summaries/hooks/use-summaries.ts
```

```tsx
const transactionsMonth = useMemo(() => {
	return transactions.filter((tx) => isWithinInterval(new Date(tx.date), interval(startOfMonth, endOfMonth)));
	/* eslint-disable-next-line react-hooks/exhaustive-deps */
}, [recalcMonthKey, transactions.length]);
```

- **Линтер хочет:** `[startOfMonth, endOfMonth, transactions]`
- **Автор использует:** `recalcMonthKey` (строка `month_2026-03`) как proxy для `startOfMonth`/`endOfMonth`
- **Решение:** Заменить `transactions.length` на `transactions`, `recalcMonthKey` — на `startOfMonth`, `endOfMonth`.
  Если perf действительно критичен — обернуть `transactions` в `useMemo` выше по цепочке, чтобы ref менялся
  только при реальных изменениях.

### D2. `use-summaries.ts:90`

```tsx
const transactionsYear = useMemo(() => {
	return transactions.filter((tx) => isWithinInterval(new Date(tx.date), interval(startOfYear, endOfYear)));
	/* eslint-disable-next-line react-hooks/exhaustive-deps */
}, [recalcYearKey, transactions.length]);
```

- Аналогично D1.

### D3. `use-calendar-txs.ts:31`

```
src/screens/transactions/txs-calendar/month/calendar-entry/hooks/use-calendar-txs.ts
```


- **Линтер хочет:** `[start, end, transactions]`

### D4. `use-backfill-rates.tsx:139`

```
src/hooks/setup/use-backfill-rates.tsx
```

```tsx
useEffect(() => {
    if (!newDates.length) return;
```

## F. Derived key как proxy dep

**Суть проблемы:** Вместо зависимости от сырого объекта (Date, etc.) используется производная строка
как триггер пересчёта — чтобы `useMemo` не срабатывал на каждый `new Date()` c тем же значением.

### F1–F2. `use-tx-dates-range.ts:42` и `:69`

```
src/hooks/use-transactions/db-queries/use-tx-dates-range.ts
```

```tsx
// useGetMinMonthDate
const minDbTxsDateResult = useMemo(() => {
	const dateTxt = minDbTxsDate?.[0]?.minDate;
	return dateTxt ? startOfMonth(dateTxt) : undefined;
	/* eslint-disable-next-line react-hooks/exhaustive-deps */
}, [minDbTxsDate, timeMode]);
```

- **`timeMode`** присутствует в deps, но не используется в теле memo — служит триггером пересчёта
  (при смене timeMode меняется запрос → меняется `minDbTxsDate`, но линтер не видит связь).
- **Решение:** `timeMode` можно убрать из deps — `minDbTxsDate` уже реагирует на смену `timeMode`
  через `useLiveQuery`. Если `minDbTxsDate` — стабильная ссылка от `useLiveQuery`, то deps корректны
  и нужно просто убрать suppress.

### F3. `use-categories.ts:147`

```tsx
const formattedDate = useMemo(() => {
	return format(transactions.dates.dayRaw, 'dd');
	/* eslint-disable-next-line react-hooks/exhaustive-deps */
}, [transactions.dates.dayFormatted]);
```

- **Тело читает:** `dayRaw` (Date)
- **Deps указывают:** `dayFormatted` (string `"2026-03-14"`)
- **Зачем:** `dayRaw` — `new Date()`, ссылка меняется каждый рендер. `dayFormatted` — строка,
  меняется только при реальной смене дня.
- **Решение:** Стабилизировать `dayRaw` через `useMemo` в `use-summaries.ts`, чтобы ref менялся
  только при реальной смене значения. Тогда можно зависеть от `dayRaw` напрямую.

## H. Reset state on ID change

### H1. `use-comment.ts:21`

```
src/screens/transactions/tx-detailed-view/components/meta/hooks/use-comment.ts
```

```tsx
const useComment = ({ id, comment }: Props) => {
    const [note, setNote] = useState(comment ?? '');

    useEffect(() => {
        setNote(comment ?? '');
        /* eslint-disable-next-line react-hooks/exhaustive-deps */
    }, [id]);
```

- **Линтер хочет:** `[id, comment]`
- **Намерение автора:** Сбрасывать `note` только при навигации на другую транзакцию (`id` сменился).
  Не сбрасывать при изменении `comment` — пользователь может редактировать поле.
- **Проблема:** Если `comment` обновится извне (например, синк) при том же `id` — `note` останется stale.
- **Решения (в порядке предпочтения):**
  1. Использовать `key={id}` на компоненте-родителе — React пересоздаст компонент при смене ID,
     `useState(comment ?? '')` получит актуальное начальное значение. Удалить `useEffect` целиком.
  2. Добавить `comment` в deps — если внешние обновления comment важнее UX сохранения локальных правок.

## J. Theme deps narrowing

### J1. `system.tsx:44`

```
src/screens/settings/components/system/system.tsx
```

```tsx
const color = useMemo(() => {
	switch (notification.status) {
		case RESULTS.DENIED:
			return theme.accents.pink;
		case RESULTS.GRANTED:
			return theme.accents.green;
		case RESULTS.LIMITED:
			return theme.accents.yellow;
		default:
			return theme.text.tertiary;
	}
	/* eslint-disable-next-line react-hooks/exhaustive-deps */
}, [theme.is_oled, theme.tint, notification.status]);
```

- **Тело читает:** `theme.accents.pink`, `theme.accents.green`, `theme.accents.yellow`, `theme.text.tertiary`
- **Deps указывают:** `theme.is_oled`, `theme.tint`, `notification.status`
- **Зачем:** `theme` — объект, ссылка меняется часто. Автор знает, что цвета акцентов зависят только
  от `is_oled` и `tint`, и использует их как proxy.
- **Решение:** Заменить на прямые deps от используемых значений:
  ```tsx
  [theme.accents.pink, theme.accents.green, theme.accents.yellow, theme.text.tertiary, notification.status];
  ```
  Или, если styled-components theme слишком нестабилен, вынести нужные цвета в переменные перед memo
  и зависеть от них.
